# 클로드 코드를 이용해 스프링 앱 개발하기 - Live 3 정리

원본 영상: https://www.youtube.com/watch?v=o74VJbF4Gkg

## 1. 전체 주제

이번 라이브의 핵심 주제는 `AI와 함께 개발할 때의 작업 흐름(workflow)`입니다.  
발표자는 단순히 "AI에게 만들어 달라"고 요청하는 방식은 결과가 평범해지기 쉽고, 좋은 결과를 얻으려면 사람도 작업 순서와 맥락을 더 잘 설계해야 한다고 설명합니다.

핵심 메시지는 다음과 같습니다.

- AI 개발도 결국은 `탐색 → 계획 → 구현 → 검토/커밋` 흐름이 중요하다.
- 좋은 결과는 짧은 프롬프트보다 `좋은 워크플로우`에서 나온다.
- AI의 성능을 끌어내려면 스킬, 플러그인, 하네스 같은 보조 수단이 필요하다.

## 2. AI 개발에서 워크플로우가 중요한 이유

발표자는 전통적인 개발과 AI 개발이 완전히 다르지는 않지만, AI 개발에서는 특히 `준비 단계`의 비중이 더 커졌다고 봅니다.

- 그냥 바로 코드 생성을 시키면 결과물이 평범하거나 기대와 어긋나기 쉽다.
- AI는 한 번에 많은 코드를 만들기 때문에, 잘못된 방향으로 가면 수정 비용도 커진다.
- 따라서 구현보다 앞선 `탐색`과 `계획`에 시간을 많이 써야 한다.

즉, AI 시대에는 "코드를 빨리 쓰는 능력"보다 `문제를 잘 자르고, 계획을 잘 세우고, 리뷰를 잘하는 능력`이 더 중요해졌다는 관점입니다.

## 3. 추천 워크플로우: 탐색 → 계획 → 구현 → 커밋

발표자가 가장 강조한 기본 흐름은 아래 4단계입니다.

### 3.1 탐색

- 현재 프로젝트 구조를 먼저 분석시킨다.
- 관련 문서, 도메인 코드, API 코드 등 필요한 범위를 지정해 이해시키는 것이 좋다.
- 새 프로젝트라면 아이디어 탐색이나 리서치부터 시작할 수 있다.

### 3.2 계획

- 바로 구현하지 말고 먼저 계획을 세우게 한다.
- 플랜 모드 등을 사용해서 파일 수정 없이 설계와 순서를 먼저 정리한다.
- 사람이 그 계획을 읽고 수정, 보완, 삭제를 반복해야 한다.

### 3.3 구현

- 계획이 충분히 정리된 뒤 구현을 시작한다.
- 한 번에 너무 크게 만들기보다 MVP나 작은 단위로 나눠서 진행하는 것이 좋다.
- 버그 수정도 바로 고치게 하기보다, 원인 분석과 수정 계획을 먼저 잡는 흐름이 더 낫다고 설명한다.

### 3.4 커밋

- 한 사이클이 끝나면 커밋 가능한 단위로 정리한다.
- 이후 문제가 생겨도 롤백 가능한 지점이 생긴다.
- 발표자는 실제로는 `검증`의 의미가 더 중요하다고도 말한다.

## 4. 초반 리서치와 브레인스토밍 방식

새 프로젝트를 시작할 때는 바로 구현보다 `리서치 → 아이디어 정리 → 스펙 작성` 흐름을 권장합니다.

예시로 발표자는 블로그 서비스나 URL 쇼트너 같은 아이디어를 AI에게 먼저 조사시키는 방식을 설명합니다.

- 경쟁 서비스 조사
- 주요 기능 비교
- 장단점 분석
- 어떤 방향으로 차별화할지 정리

그 후에야 스펙과 구현 계획으로 넘어갑니다.  
즉, AI를 단순 코드 생성기가 아니라 `리서처 + 설계 보조자`로 먼저 활용하는 방식입니다.

## 5. 터미널 환경: `Zed?`가 아니라 `Zmux/Smux 계열 터미널` 활용 경험

발표자는 메인 터미널 환경을 바꾸었고, 현재 사용하는 도구의 장점을 길게 설명합니다.  
핵심은 `AI 코딩에 적합한 멀티 워크스페이스/멀티 페인 환경`입니다.

주요 장점:

- 워크스페이스별로 작업을 분리할 수 있다.
- 한 화면에서 여러 프로젝트/세션을 동시에 운영할 수 있다.
- 페인 단위 분할이 편해서 AI 응답 대기 중 다른 작업을 병행하기 좋다.
- 브라우저를 같은 터미널 환경 안에서 열어 웹 테스트와 AI 작업을 함께 볼 수 있다.
- 작업 완료나 대기 상태를 시각적으로 알림 받아 멀티태스킹이 쉽다.

발표자는 AI 코딩에서는 기다리는 시간이 많기 때문에, `동시에 여러 작업을 돌릴 수 있는 작업 환경`이 매우 중요하다고 봅니다.

## 6. 최신 기술 정보는 AI가 기본적으로 모를 수 있음

발표자는 스프링 부트, 자바, Next.js 같은 기술의 최신 버전을 AI가 틀리게 아는 경우를 예로 듭니다.

핵심 포인트:

- AI 모델은 학습 시점 이후 최신 정보를 모를 수 있다.
- 그래서 "최신 버전으로 개발해 줘"라고만 하지 말고,
  - 검색하게 하거나
  - 공식 문서를 읽게 하거나
  - 관련 지식을 스킬로 정리해 재사용하게 해야 한다.

즉, 최신성을 요구하는 작업에서는 `검색 + 문서 학습 + 스킬화`가 필요하다는 설명입니다.

## 7. 스킬(Skill)의 역할과 활용법

이번 영상에서 가장 자주 반복된 주제 중 하나가 `스킬`입니다.

발표자는 스킬을 크게 두 용도로 설명합니다.

### 7.1 작업 자동화용 스킬

- 자주 반복하는 긴 프롬프트를 짧은 명령으로 바꾼다.
- 예: 프로젝트 생성, 특정 워크플로우 실행, 리뷰 요청, 계획 수립

### 7.2 레퍼런스/지식 저장용 스킬

- AI가 잘 모르는 기술 지식
- 팀 내부 규칙
- 특정 프로젝트의 중요한 컨텍스트
- 특정 버전 차이점

이런 내용을 스킬로 정리해두면, 새 세션에서도 바로 불러 쓸 수 있습니다.

발표자의 관점:

- 스킬은 개인용으로도 중요하지만 팀 공유용으로도 중요하다.
- 다만 모델 버전이 바뀌면 스킬의 가치가 줄거나 오히려 방해될 수 있으니 계속 평가해야 한다.

## 8. 플러그인 추천과 사용 기준

발표자는 여러 플러그인을 언급하지만, 무조건 많이 깔기보다 목적에 맞게 쓰는 쪽을 선호합니다.

### 언급된 주요 플러그인/도구

- Claude 공식 플러그인 마켓플레이스
- Superpowers
- Ralph
- Firecrawl
- 개인 제작 플러그인들

### 발표자의 사용 기준

- 처음 시작하는 사람은 너무 많은 자동화 플러그인에 의존하지 말 것
- 먼저 공식 문서와 기본 워크플로우를 이해할 것
- 이후 반복 작업이 생기면 스킬/플러그인으로 확장할 것

## 9. Superpowers와 Ralph의 차이

발표자는 본인의 주요 도구 3가지를 다음처럼 정리합니다.

- `플랜 모드`: 가장 가볍고 빠른 방식
- `Superpowers`: 큰 기능, 초기 개발, 중요한 설계 작업에 적합
- `Ralph`: 좀 더 빠르고 가벼운 확인/정리 작업에 적합

### Superpowers의 특징

- 브레인스토밍 단계가 강력하다.
- 질문을 통해 요구사항을 구체화한다.
- 선택지를 제시하고 고르게 한다.
- 비주얼 컴패니언을 통해 화면 중심 설계를 돕는다.
- 스스로 리뷰하고, 구현 계획까지 구체적으로 만든다.

### Ralph의 특징

- 질문은 적지만 빠르다.
- 너무 큰 작업이 아닌 경우 효율적이다.
- PRD나 비교적 빠른 흐름의 정리에 유용하다.

## 10. 실제 예시: URL 쇼트너 + 애널리틱스 서비스 기획

영상 후반부에서는 예시 프로젝트로 `URL 쇼트너 + 애널리틱스`를 선택합니다.

이 과정에서 AI는 다음을 수행합니다.

- 목적 확인: 학습용인지, 실서비스용인지, 포트폴리오용인지
- 대상 범위 확인: 개인용인지 팀용인지 SaaS 공개용인지
- 핵심 분석 기능 범위 정의
- 인증 방식 정의
- DB 선택
- 커스텀 도메인 여부
- 프론트엔드 포함 여부
- 빌드 도구 선택
- QR 코드 생성 여부
- 아키텍처 옵션 제안

여기서 발표자가 강조한 포인트는:

- 처음부터 모든 기능을 넣지 말고 MVP 범위로 좁힐 것
- 질문을 통해 요구사항을 좁히는 과정 자체가 중요하다는 것

## 11. 비주얼 컴패니언과 설계 리뷰

비주얼 컴패니언은 텍스트가 아니라 브라우저 기반 시각 자료로 설계 내용을 보여주는 기능으로 소개됩니다.

장점:

- 아키텍처를 도식으로 보기 쉽다.
- 텍스트보다 빠르게 이해할 수 있다.
- 설계 검토 시 인지 부하를 줄인다.

하지만 발표자는 AI가 제안한 설계를 그대로 받아들이지 말고, 아래 항목을 사람이 적극적으로 검토해야 한다고 말합니다.

- 패키지 구조
- 폴더 구조
- 기술 선택
- 테스트 방식
- API 버전 전략
- 예외 처리 방식

## 12. 멀티 모델 리뷰: Claude + Codex + Gemini

중요한 설계나 계획은 Claude 혼자만 보지 않고 다른 모델에게도 리뷰를 받는 방식을 보여줍니다.

핵심 방식:

- 별도 페인에 Codex, Gemini를 띄운다.
- 같은 설계 문서를 각 모델에 전달한다.
- 결과를 파일로 받아 모은다.
- Claude가 그 의견을 다시 종합하게 한다.

의도:

- 자기검토에서 놓친 문제를 다른 모델이 찾아낼 수 있다.
- 모델별 시각 차이를 활용할 수 있다.
- 최종 결정은 사람이 하되, Claude가 컨센서스를 정리하게 할 수 있다.

발표자는 특히 `리뷰` 영역에서 멀티 모델 접근이 유용하다고 봅니다.

## 13. 퍼미션, 보안, 위험 작업 차단

영상에서 꽤 비중 있게 다룬 주제가 `AI에게 어디까지 권한을 줄 것인가`입니다.

### 퍼미션 설정

- 자주 쓰는 명령은 허용 목록에 넣어 확인 횟수를 줄일 수 있다.
- 단, 업무 환경에서는 무조건 자동 허용을 늘리는 것은 위험할 수 있다.

### 디나이 목록

- 민감 파일 읽기 금지
- 위험한 수정 금지
- 파괴적 명령 금지

예시:

- SSH 키
- AWS 자격 증명
- `.env` 내부 민감 정보
- `rm -rf` 류 명령

### 더 강력한 보호: 훅(hook)

발표자는 allow/deny만으로는 충분하지 않을 수 있다고 보고,  
실행 직전에 차단 로직을 넣는 `hook` 방식이 더 강력하다고 설명합니다.

즉, 보안은 프롬프트 수준이 아니라 `실행 전 코드 레벨 차단`이 더 믿을 만하다는 입장입니다.

## 14. 컨텍스트 관리의 중요성

발표자는 컨텍스트가 커질수록 AI가 오히려 멍청해질 수 있다고 강조합니다.

핵심 요지:

- 긴 대화가 계속 누적되면 AI가 현재 중요한 문제에 집중하지 못한다.
- 앞의 계획, 대화, 실험이 전부 따라붙으면서 오히려 효율이 떨어진다.
- 그래서 적절한 작업 단위로 세션을 끊고, 다음 세션에 필요한 내용을 문서로 남겨야 한다.

권장 방식:

- 1~2 사이클 단위로 세션 정리
- 다음 세션에 필요한 요약 문서 생성
- 필요한 스킬/메모리만 다시 로드

## 15. 하네스(Harness)의 개념

질문 답변 과정에서 발표자는 `하네스`를 다음처럼 설명합니다.

- AI 모델 자체는 도구를 직접 쓰지 못한다.
- Claude Code는 Claude 모델 바깥에 입혀진 실행 프레임워크다.
- 즉, Claude Code 자체가 하나의 하네스다.
- 여기에 스킬, 플러그인, 훅, 메모리, 자동화 스크립트 등을 더 얹으면 `나만의 하네스`가 된다.

정리하면:

- `Claude 모델` = 두뇌
- `Claude Code` = 기본 하네스
- `스킬/플러그인/훅/자동화` = 사용자 맞춤 확장 하네스

발표자는 AI 개발 실력이란 결국 `좋은 하네스를 설계하고 운영하는 능력`까지 포함한다고 봅니다.

## 16. AI가 100% 코딩해도 개발 책임은 사람에게 있음

매우 중요한 메시지 중 하나입니다.

- 개인 프로젝트는 100% Claude Code로 코딩할 수 있다.
- 실제 회사에서도 대부분의 코드 생성을 AI가 맡는 경우가 늘고 있다.
- 하지만 그렇다고 개발 책임까지 AI에 넘겨지는 것은 아니다.

발표자가 강하게 말한 부분:

- 이해하지 못한 코드를 PR로 올리는 건 문제다.
- "AI가 만들었으니 저도 잘 모릅니다"는 좋은 개발 방식이 아니다.
- AI가 코드를 만들더라도 사람은 설계, 판단, 리뷰, 책임을 져야 한다.

즉, `코드는 AI가 써도 개발은 사람이 한다`는 것이 핵심 입장입니다.

## 17. 대화 기록과 작업 맥락도 함께 공유해야 함

발표자는 AI 시대에는 코드만 공유하는 것이 충분하지 않을 수 있다고 봅니다.

그래서 자신이 만든 스킬을 통해 아래 정보를 함께 저장하려고 합니다.

- 어떤 프롬프트를 사용했는지
- 어떤 AI 응답이 있었는지
- 어떤 도구를 썼는지
- 어떤 파일이 바뀌었는지
- 어떤 스킬이 개입했는지

의도는 다음과 같습니다.

- 코드 리뷰 시 맥락을 함께 볼 수 있게 하기
- 협업자가 변경 의도를 더 쉽게 이해하게 하기
- AI와의 상호작용 자체를 개발 산출물로 다루기

즉, 앞으로는 `코드 + 설계 문서 + 대화 맥락`이 함께 공유되어야 한다는 문제의식입니다.

## 18. 팀 단위/조직 단위 도입 시 고민

개인 생산성을 넘어서 팀 단위로 AI 개발을 도입하면 더 어려운 문제가 생긴다고 말합니다.

- 팀 공통 스킬을 어떻게 관리할 것인가
- 모델 버전이 바뀌면 무엇을 다시 평가할 것인가
- 회사 코딩 규칙과 AI 워크플로우를 어떻게 맞출 것인가
- 개인별 스타일 차이를 어떻게 줄일 것인가

발표자는 이 부분이 아직 완전히 정리된 상태가 아니며, 앞으로 계속 실험과 평가가 필요하다고 봅니다.

## 19. 발표자의 최종 조언

전체 내용을 관통하는 조언은 아래와 같습니다.

- 공식 문서를 먼저 읽어라.
- 바로 구현하지 말고 탐색과 계획에 시간을 써라.
- 스킬은 많이 만들되 계속 평가하고 업데이트하라.
- 최신 기술 정보는 검색과 레퍼런스로 보완하라.
- 한 번에 너무 큰 범위를 만들지 말고 작은 사이클로 나눠라.
- AI가 만든 코드라도 사람이 반드시 이해하고 검토해야 한다.
- 자기만의 워크플로우와 하네스를 꾸준히 다듬어라.

## 20. 한 줄 요약

이 영상은 단순한 "클로드 코드 사용법"이 아니라,  
`AI 코딩 시대에 개발자가 어떻게 워크플로우, 스킬, 리뷰, 보안, 협업 방식을 재설계해야 하는가`를 실전 사례 중심으로 설명한 라이브입니다.

---

## 부록 A. 발표자의 주장

아래는 영상 전반에서 반복적으로 드러난 발표자의 핵심 주장입니다.

### 1. AI 코딩의 성패는 구현보다 워크플로우에 달려 있다

- 단순히 "만들어 줘"라고 하면 평범한 결과가 나온다.
- 탐색과 계획을 먼저 잘해야 구현 품질이 올라간다.
- AI를 잘 쓰는 사람은 프롬프트 한 줄보다 `작업 순서와 범위 설정`을 더 중요하게 본다.

### 2. AI는 코드 생성기이면서 동시에 설계 보조자, 리서처, 리뷰어다

- 구현만 맡기기보다 조사, 비교, 계획, 검토에 먼저 활용하는 편이 더 효과적이다.
- 특히 초기 요구사항 정리와 대안 비교에서 큰 도움을 준다.

### 3. 좋은 결과를 위해서는 사람의 개입이 줄어드는 것이 아니라 바뀌는 것이다

- 사람이 직접 타이핑하는 양은 줄 수 있다.
- 대신 설계 검토, 범위 조절, 품질 판단, 최종 책임은 더 중요해진다.
- "AI가 만들었으니 잘 모른다"는 태도는 허용되지 않는다.

### 4. 스킬과 하네스는 생산성의 핵심 자산이다

- 반복되는 작업 흐름은 스킬로 만들어야 한다.
- 자주 쓰는 기술 지식도 스킬이나 메모리로 정리해야 한다.
- 결국 개인과 팀의 경쟁력은 `모델 자체`보다 `모델을 어떻게 감싸서 쓰는가`에서 벌어진다.

### 5. 모델은 똑똑하지만 항상 맞지 않다

- 최신 버전을 틀리게 알 수 있다.
- 기술적으로 그럴듯하지만 잘못된 설계를 제안할 수 있다.
- 자기검토만으로는 놓치는 문제가 있기 때문에 교차 리뷰가 필요하다.

## 부록 B. 실무 팁

아래 항목은 영상을 실무 관점으로 재구성한 체크리스트입니다.

### 1. 새 작업을 시작할 때

- 먼저 AI에게 현재 프로젝트를 탐색시키기
- 필요한 범위만 지정해서 읽히기
- 큰 작업이면 바로 구현하지 말고 계획 모드 먼저 사용하기
- 가능하면 MVP 범위로 줄이기

### 2. 큰 기능을 만들 때

- 브레인스토밍으로 요구사항을 질문 형태로 구체화하기
- 스펙 문서를 먼저 만들기
- 구현 계획까지 본 뒤에 코딩 시작하기
- 중간에 한 번 더 셀프 리뷰 또는 외부 모델 리뷰 받기

### 3. 작은 수정이나 버그 수정 때

- 에러 원인 분석부터 시키기
- 수정 계획을 먼저 확인하기
- 재현 테스트 또는 회귀 테스트를 함께 만들게 하기
- 너무 단순한 수정만 즉시 구현 모드로 처리하기

### 4. 최신 기술을 쓸 때

- "최신 버전 기준으로 확인해"를 명시하기
- 공식 문서/검색 결과를 읽게 하기
- 자주 쓰는 기술은 버전 차이점 스킬을 따로 만들기

### 5. 컨텍스트 관리

- 세션을 너무 길게 끌지 않기
- 큰 작업은 1~2 사이클 단위로 끊기
- 다음 세션용 요약 문서를 남기기
- 오래된 대화가 계속 붙어 성능을 떨어뜨리지 않게 관리하기

### 6. 팀 협업 시

- 개인 스킬과 팀 공용 스킬을 구분하기
- 공용 코딩 규칙은 메모리/스킬로 정리하기
- AI 작업 결과물에 설계 문서와 의도도 같이 남기기
- 중요한 작업은 사람 리뷰를 반드시 거치기

### 7. 멀티 모델 활용 시

- 메인 구현 모델 1개를 정하고
- 설계 리뷰, 반론 제기, 검토 전용 모델을 따로 둘 수 있다
- 단, 의견이 갈릴 때 최종 의사결정 기준은 사람이 가져야 한다

## 부록 C. 주의사항

영상 내용을 실무적으로 해석했을 때 특히 주의해야 할 부분들입니다.

### 1. 설계를 AI에 통째로 맡기지 말 것

- AI가 그럴듯한 구조를 만들더라도 현재 시스템 문맥과 안 맞을 수 있다.
- 패키지 구조, 테스트 전략, 기술 선택은 사람이 검토해야 한다.

### 2. 큰 범위를 한 번에 구현시키지 말 것

- AI는 한 번에 많이 만들 수 있지만, 그만큼 잘못될 때 피해도 커진다.
- 큰 기능일수록 단계별로 쪼개고 중간 검증 지점을 둬야 한다.

### 3. 최신성 착각을 경계할 것

- 스프링, 자바, 프론트엔드 프레임워크 버전을 쉽게 틀릴 수 있다.
- "그럴듯한데 오래된 답"이 가장 위험한 유형이다.

### 4. 보안 파일 접근을 허술하게 두지 말 것

- `.env`, SSH 키, 클라우드 자격증명은 원천 차단이 우선이다.
- 허용 규칙만 믿지 말고 훅 수준 차단도 고려해야 한다.

### 5. 허용 규칙을 무분별하게 늘리지 말 것

- 편하다고 모두 allow 하면 업무 환경에서는 사고 가능성이 커진다.
- 자주 쓰는 안전한 명령만 선별적으로 올리는 편이 낫다.

### 6. AI가 만든 결과를 이해 없이 머지하지 말 것

- 코드 리뷰에서 설명할 수 없는 코드는 팀 자산이 되기 어렵다.
- 이해 안 되는 부분은 AI에게 다시 설명받고, 필요하면 문서화해야 한다.

### 7. 스킬도 계속 평가해야 할 것

- 모델이 바뀌면 스킬이 불필요해질 수 있다.
- 심하면 오히려 성능을 떨어뜨릴 수도 있다.
- "한 번 만든 스킬"이 아니라 "계속 평가하는 자산"으로 다뤄야 한다.

## 부록 D. 실무 적용용 압축 정리

영상 내용을 실무에서 바로 적용하려면 아래 순서가 가장 현실적입니다.

1. 공식 문서와 기본 워크플로우부터 이해한다.
2. 탐색 → 계획 → 구현 → 검토 흐름을 습관화한다.
3. 반복되는 프롬프트와 지식은 스킬로 만든다.
4. 최신 기술은 검색과 공식 문서로 보강한다.
5. 큰 작업은 작게 나누고 중간 리뷰를 넣는다.
6. 중요한 설계는 다른 모델이나 사람에게도 검토받는다.
7. 보안 파일과 위험 명령은 도구 수준에서 차단한다.
8. 결과 코드뿐 아니라 설계 의도와 대화 맥락도 남긴다.
