# (56) 클로드 코드를 이용해 스프링 앱 개발하기 - Live 4 정리

- 영상: https://www.youtube.com/watch?v=_FlVon9Nyos
- 주제: Claude Code로 Spring 기반 앱을 개발할 때의 실전 워크플로우, `Ralph`/`PRD`/`Ralph Loop` 활용법, 스킬과 프롬프트 운영 방식

## 한줄 요약

이번 라이브는 이전 시간에 만들던 짧은 URL 서비스 프로젝트를 이어서 다듬으면서, Claude Code로 작업할 때 `탐색 -> 계획 -> 구현 -> 검증/커밋` 흐름을 어떻게 유지할지와 `Ralph` 계열 도구를 어떻게 실전에 적용할지를 보여주는 내용이다.

## 핵심 워크플로우

발표자가 반복해서 강조한 기본 흐름은 다음과 같다.

1. `탐색`
   프로젝트를 처음 연 AI는 현재 구조와 상태를 모르므로 먼저 코드베이스를 분석해야 한다.
2. `계획`
   바로 구현하지 말고, 작업 범위와 완료 조건을 먼저 정리한다.
3. `구현`
   계획에 따라 기능을 작게 나눠 진행한다.
4. `검증`
   테스트, UI 확인, 인수 조건 점검 등을 통해 결과를 확인한다.
5. `커밋`
   한 사이클이 끝나면 작업 단위를 기록 가능한 상태로 남긴다.

즉, AI에게 바로 "만들어줘"라고 던지기보다 먼저 현재 상태를 읽히고, 계획을 세우게 한 뒤, 작은 단위로 구현시키는 것이 중요하다는 설명이다.

## 이번 라이브의 실습 대상

- 이전 시간에 만들던 `Short URL` 서비스 계속 개발
- 멀티 유저 사용
- 구글 소셜 로그인
- 랜딩 페이지 개선
- 링크 통계 기능 추가
- 사이드바 내비게이션 추가

실습 중에는 통계 페이지와 사이드바를 추가하는 기능을 예시로 사용했다.

## Claude Code 사용 팁

### 1. 권한 확인 생략 옵션

발표자는 장시간 작업 시 권한 확인이 반복되는 것이 번거롭기 때문에 `dangerously skip permissions` 같은 옵션을 사용하기도 한다고 설명했다.  
다만 이는 매우 위험할 수 있으며, 운영 서버 연결이나 민감한 도구가 없는 제한된 환경에서만 조심스럽게 써야 한다고 언급했다.

### 2. 상태 표시 HUD

기본 상태 표시보다 더 많은 정보를 보여주는 HUD를 사용하면 다음을 확인하기 좋다고 설명했다.

- 현재 어떤 스킬이 실행 중인지
- 백그라운드 에이전트가 몇 개 돌아가는지
- 각 에이전트가 무슨 작업을 하는지
- 사용량/리밋 상태

다만 초보자에게는 너무 복잡할 수 있으므로 처음부터 무거운 확장 환경을 쓰기보다 기본 사용법부터 익히는 것을 권장했다.

### 3. 스킬 자동 호출은 제어가 필요함

특정 스킬이 자동 호출되면 편하지만, 원하지 않는 상황에서 시간이 오래 걸릴 수도 있다.  
이 경우에는 해당 스킬을 사용하지 말라고 명시적으로 프롬프트에 적어 제어하면 된다고 설명했다.

## 계획 수립 도구 비교

발표자는 상황에 따라 세 가지 접근을 구분해서 사용한다고 설명했다.

### 1. 기본 Plan Mode

- 간단한 계획 수립에 적합
- 비교적 빠르게 진행 가능

### 2. Superpowers의 Brainstorming

- 새 기능이 크거나 복잡할 때 적합
- 요구사항이 불명확할 때 좋음
- 질문과 선택지를 여러 단계에 걸쳐 주면서 스펙을 구체화함
- 프로젝트 초기 설계, 리팩터링, 큰 기능 추가에 유용

### 3. Ralph의 PRD

- Brainstorming보다 가볍고 빠름
- 중간 정도 복잡도의 작업에 적합
- 몇 가지 핵심 질문만 던진 뒤 PRD 문서를 생성함
- 요구사항을 `사용자 스토리`와 `인수 조건` 중심으로 정리해 주는 점이 장점

즉, 아주 단순하면 Plan Mode, 크고 복잡하면 Brainstorming, 그 중간 지점이면 Ralph PRD를 쓰는 전략이다.

## Ralph PRD의 특징

`/prd`로 요청하면, AI가 기능 요구사항을 바로 구현하지 않고 먼저 문서화한다.

주요 특징:

- 필요한 통계 항목, 페이지 구조, 차트 종류 같은 질문을 한 번에 묶어서 확인
- 결과를 사용자 스토리 형태로 정리
- 각 스토리마다 `완료 조건(acceptance criteria)`를 포함
- 기능 요구사항과 범위 외 항목까지 구분해서 적어 줌

발표자는 이 방식이 좋은 이유로, "그냥 통계 기능 넣어줘" 같은 모호한 요청을 비교적 짧은 상호작용만으로 구조화해 준다는 점을 들었다.

## Ralph의 역할

PRD가 사람 친화적인 문서라면, `Ralph`는 그것을 AI가 반복 작업하기 좋은 구조로 바꿔 주는 단계로 소개되었다.

핵심 아이디어:

- PRD를 구조화된 `JSON` 형태로 변환
- 사용자 스토리를 더 잘게 쪼갤 수 있음
- 각 스토리마다 완료 여부를 `true/false`로 관리
- AI가 다음에 무엇을 해야 하는지 판단하기 쉬워짐

발표자는 AI가 자연어도 잘 이해하지만, `JSON`이나 `XML`처럼 구조화된 형식으로 주면 훨씬 더 명확하게 동작한다고 설명했다.

## Ralph Loop의 개념

`Ralph Loop`는 "완료될 때까지 반복 실행"을 자동화하는 방식이다.

동작 방식은 단순하다.

- 프롬프트를 준다
- 작업을 수행한다
- 완료 조건을 점검한다
- 완료되지 않았으면 다시 같은 작업을 반복시킨다
- 조건을 만족하면 종료한다

즉, 본질적으로는 `완료 조건 기반 루프`다.

## Ralph Loop 사용 방식 2가지

### 1. 쉘 스크립트 방식

- Claude Code를 한 번 실행하고 종료하는 식으로 반복
- 매 반복마다 새 세션으로 시작
- 컨텍스트를 매번 초기화하는 방식에 가깝다

### 2. 플러그인 방식

- Claude Code 안에서 반복
- 컨텍스트를 유지한 채 다음 반복 수행
- 작업 흐름을 계속 보기에 더 편함

발표자는 개인적으로 쉘 스크립트 방식보다 플러그인 방식을 선호한다고 말했다.  
이유는 작업 흐름이 더 잘 보이고, 현재 컨텍스트가 유지되는 편이 실무에서 유리한 경우가 많기 때문이다.

## 실제 루프 운영 방법

발표자가 보여준 방식은 대략 다음 순서다.

1. PRD 생성
2. Ralph로 JSON 변환
3. `prompt.md` 같은 작업 지시 파일 준비
4. 아직 완료되지 않은 사용자 스토리 중 우선순위가 높은 것을 진행하도록 지시
5. 완료 시 해당 스토리의 `pass` 값을 `true`로 변경
6. 모든 스토리가 완료되면 `COMPLETE` 같은 종료 문자열 출력
7. Ralph Loop가 이 종료 조건을 보고 반복 중단

중요한 포인트는, AI가 길게 작업하다 중간에 멈추는 문제를 이런 루프가 보완해 준다는 점이다.

## Ralph Loop가 특히 유용한 경우

발표자는 다음 같은 경우를 예로 들었다.

- 테스트 커버리지가 90% 이상 될 때까지 테스트 추가 반복
- 복잡한 문제를 여러 번 시도하며 개선해야 할 때
- 사용자 스토리 여러 개를 끝까지 밀어야 할 때
- AI가 중간에 멈추는 현상을 강제로 넘기고 싶을 때

즉, 단발성 요청보다 `조건 만족까지 반복`이 핵심인 작업에 잘 맞는다.

## 한계와 주의점

### 1. 같은 프롬프트를 반복하는 한계

기본 루프는 같은 프롬프트를 반복해서 던진다.  
그래서 중간에 새롭게 얻은 통찰을 다음 시도에 반영하지 못할 수 있다.

이를 보완하려면:

- 이전 시도 결과를 바탕으로 프롬프트를 개선하는 단계 추가
- 루프를 돌면서 프롬프트 자체를 점점 업그레이드하는 방식 사용

### 2. 작업 단위를 작게 쪼개야 함

기능 하나를 너무 크게 주면:

- 구현 도중 멈춤
- UI는 생겼지만 연결이 안 됨
- 일부 기능만 되고 마무리가 안 됨

그래서 화면만 먼저, 로직만 먼저, 테스트만 먼저 등으로 더 잘게 나눌 필요가 있다고 설명했다.

### 3. 자동 커밋 주의

Ralph 계열 흐름에서는 사용자 스토리 하나가 끝날 때마다 커밋을 남길 수 있다.  
전체 기능 단위로 커밋하고 싶다면 중간 커밋을 하지 말라고 명시해 두는 것이 좋다고 조언했다.

## 실습 결과

통계 페이지와 사이드바 추가 작업을 루프로 돌린 뒤 실제 서버를 다시 실행해서 확인했다.

확인 과정에서 드러난 점:

- 사이드바와 통계 화면은 생성됨
- 통계 항목으로 총 클릭 수, 유니크 클릭 수 등이 표시됨
- 차트 기반 시각화도 추가됨
- 다만 처음 기대한 위치와 실제 UI 배치는 달라서, 사용자가 직접 다시 확인하고 수정 지시를 주는 과정이 필요했음

즉, AI가 많은 것을 빠르게 만들 수는 있지만, 최종 UI/동작 검수는 여전히 사람이 해야 한다는 점을 보여준다.

## Spring 개발에 대한 조언

발표자는 Claude Code가 Spring을 "적당히 그럴듯하게" 만들어 줄 수는 있지만, 팀이나 개발자가 원하는 아키텍처/스타일/규칙을 자동으로 완벽히 지키지는 못한다고 설명했다.

그래서 필요한 것은:

- 초반에 프로젝트 규칙을 잘 정리해 두기
- 새로운 라이브러리/기술은 먼저 학습시키기
- 자주 쓰는 지식은 스킬로 만들어 재사용하기
- 자신만의 개발 방식과 패턴을 문서화해 두기

## 왜 디테일이 자꾸 빠지는가

발표자는 AI 코딩에서 결과가 엉성해지는 이유를 이렇게 설명했다.

- 개발자는 물론 비개발자도 "완성된 제품에 필요한 디테일"을 자주 놓친다
- 요구사항을 충분히 상세하게 떠올리지 못하면 결과도 평범하거나 유치해진다
- 그래서 계획 단계에서 AI의 도움을 받아 빠진 요구사항을 끌어내는 과정이 중요하다

이 때문에, 좋은 바이브 코딩은 단순히 "코드를 많이 생성하는 것"이 아니라 `요구사항을 더 잘 정의하는 능력`과 연결된다고 강조했다.

## 스킬 활용 조언

- 공개 스킬을 바로 맹신하지 말고 먼저 분석해서 이해할 것
- 원하는 동작과 다르면 수정해서 자신만의 버전으로 만들 것
- 스킬 관리 인터페이스는 아직 불편하므로 직접 운영 감각을 익혀야 함
- 자주 쓰는 패턴은 플러그인/스킬로 정리해 두면 생산성이 올라감

## 프롬프트 엔지니어링 관련 포인트

발표자는 최종적으로 스킬도 플러그인도 결국 프롬프트를 더 잘 구성하는 방식이라고 설명했다.

특히 강조한 내용:

- 중요한 요구사항은 더 명확한 표현으로 작성
- 반드시 지켜야 할 조건은 강하게 표시
- 같은 핵심 지시를 프롬프트 앞/중간/끝에 반복 배치하는 기법도 유용
- 어떤 표현이 AI에게 더 잘 먹히는지 계속 실험해야 함

즉, AI 코딩을 잘하려면 도구만이 아니라 프롬프트 작성 역량도 계속 함께 키워야 한다는 메시지다.

## 전체 결론

이번 라이브의 핵심 메시지는 다음과 같다.

- AI에게 바로 구현시키기보다 `탐색 -> 계획 -> 구현 -> 검증` 흐름을 지켜라
- 작업 크기에 따라 `Plan Mode`, `Brainstorming`, `Ralph PRD`를 구분해 써라
- 장시간 작업이나 중간 멈춤 문제는 `Ralph Loop`로 보완할 수 있다
- 구조화된 요구사항과 명확한 완료 조건이 결과 품질을 크게 좌우한다
- 결국 성능 차이는 도구 자체보다도 `좋은 요구사항`, `작은 작업 단위`, `지속적인 검수`에서 나온다

## 참고 메모

- 발표자는 추후 `에이전틱 코딩` 관련 얇은 책을 집필할 계획이라고 언급했다.
- 장기적으로는 자신만의 스킬, 프롬프트, 개발 규칙을 축적하는 것이 중요하다는 점을 다시 강조했다.
